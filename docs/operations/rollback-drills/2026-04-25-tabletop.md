### Step 4 — advise sideload users (target 15-18 min)

**FINDING (gap §G3, HIGH):** the playbook's daemon-restart command is

    pkill -f 'forge-daemon' && /path/to/target/release/forge-daemon &

`pkill -f 'forge-daemon'` matches anything with that substring in its
full command line. During this drill, `pgrep -f 'forge-daemon'` from a
shell whose `cwd` was `/mnt/colab-disk/DurgaSaiK/forge/forge` matched
**unrelated shell processes** (PIDs 2421410 + 3697418) because their
ancestry contained the directory path. A real rollback running this
command would kill those processes — the same hazard the project
CLAUDE.md warns against (`stop using killall! only kill the dev server
selectively!`).

**Recommended replacement** (forge-daemon writes a pidfile at
`$FORGE_DIR/forge.pid`; path computed by `forge_core::default_pid_path`;
the lock is acquired by `main.rs::acquire_pid_lock`):

    DAEMON_PID=$(cat "${FORGE_DIR:-$HOME/.forge}/forge.pid" 2>/dev/null || true)
    if [ -n "$DAEMON_PID" ] && kill -0 "$DAEMON_PID" 2>/dev/null; then
        kill "$DAEMON_PID"
        # Wait briefly for graceful shutdown.
        for _ in 1 2 3 4 5; do
            kill -0 "$DAEMON_PID" 2>/dev/null || break
            sleep 1
        done
    fi
    /path/to/target/release/forge-daemon &

This is captured as gap §G3 below.

## Discovered gaps

### G1 — `.github/pending-rollback` flag has no enforcement mechanism

Playbook Step 0 instructs operators to commit `.github/pending-rollback`
with the implication that branch protection refuses merges while it
exists. That requires a GitHub repo-settings rule
(`Settings → Branches → Branch protection rules → Require status check`)
that is *not* configured today. The flag file alone is informational.

**Fix:** either
1. add a CI step that fails when `.github/pending-rollback` exists
   (mechanical, in-repo, doesn't need GitHub settings access), OR
2. document the GitHub branch-protection rule needed and add it as a
   one-time setup step under `docs/operations/` post-2P-1-deploy.

Option 1 is cheaper and self-contained; option 2 is the real-world
enforcement.

### G2 — `gh release delete --cleanup-tag=false` is non-idiomatic

`gh release delete --help` shows `--cleanup-tag` as a bare boolean.
The form `--cleanup-tag=false` is accepted by cobra/gh and is
functionally identical to omitting the flag, but reads as if a value
is required. Cosmetic; consider rewriting to:

    # Default: keep the tag.
    gh release delete "$TAG" --yes
    # If the tag must also go:
    gh release delete "$TAG" --yes --cleanup-tag

### G3 — `pkill -f 'forge-daemon'` is unsafe (HIGH)

See Step 4 finding above. Replace with a pidfile-based kill.

### G4 — DB compatibility matrix flags an open TODO

Playbook itself notes: *"We do not currently auto-backup on migration —
2P-1b §5a TODO: add pre-migration DB snapshot."* That gap is a
genuine production safety hole when rolling back across schema
boundaries. Track as 2P-1b backlog.

### G5 — No quarterly drill calendar

Playbook says "Run this drill quarterly". No reminder mechanism. Add a
recurring cron or a HANDOFF entry that pings every 90 days.
