# Forge v0.2.0 Roadmap

## Vision: One Graph, One Engine, Everything Connected

Replace the fragmented memory architecture (4 separate systems) with a unified KuzuDB-powered knowledge graph that holds code structure, decisions, patterns, lessons, and session history in one queryable graph.

## Current State (v0.1.5)

| System | Backend | What it stores | Problem |
|--------|---------|---------------|---------|
| codebase-memory-mcp | SQLite | Code structure | Not a graph DB, no relationships |
| episodic-memory | SQLite + vectors | Conversation transcripts | Unstructured, no knowledge extraction |
| MEMORY.md | Flat file | Decisions | No relationships, no search, no temporal |
| STATE.md | Flat file | Session state | Ephemeral, no cross-session aggregation |

## Target State (v0.2.0)

**One MCP server** (forge-graph) backed by **KuzuDB** (embedded, zero dependencies) containing:

### Unified Graph Schema

```
CODE NODES (from Axon):
  Function    { name, file, line, signature, module }
  Class       { name, file, inheritance, interfaces }
  Module      { name, path, language }

CODE EDGES (from Axon):
  CALLS, EXTENDS, IMPLEMENTS, IMPORTS, TYPE_REF, CONTAINS

MEMORY NODES (new):
  Decision    { title, rationale, status, created_at, updated_at, session_id }
  Pattern     { name, description, frequency, confidence, domain }
  Lesson      { insight, context, severity, source_session }
  Preference  { key, value, scope (user|project), confidence }
  Session     { started_at, ended_at, mode, project, outcome }

MEMORY EDGES (new):
  SUPERSEDES       Decision -> Decision     (temporal: decision evolution)
  MOTIVATED_BY     Decision -> Lesson       (why a decision was made)
  IMPLEMENTS       Decision -> Pattern      (decision follows a pattern)
  CONTRADICTS      Decision -> Decision     (conflicting choices)
  AFFECTS          Decision -> Function|Class|Module  (decision impacts code)
  LEARNED_IN       Lesson   -> Session      (provenance)
  DECIDED_IN       Decision -> Session      (provenance)
  EVOLVED_FROM     Pattern  -> Pattern      (skill/pattern evolution)
```

### Unified MCP Tools

```
Code intelligence (existing Axon tools):
  axon_query            — hybrid search (BM25 + semantic + fuzzy)
  axon_context          — 360-degree symbol view
  axon_impact           — blast radius analysis
  axon_dead_code        — unreachable symbols
  axon_detect_changes   — git diff to symbol mapping
  axon_cypher           — raw Cypher queries

Memory tools (new):
  forge_remember        — store decision/pattern/lesson/preference
  forge_recall          — search by concept, type, or timeframe
  forge_link            — create relationships between any nodes
  forge_decisions       — list decisions affecting a code path
  forge_patterns        — list learned patterns for a domain
  forge_timeline        — temporal evolution of any entity
  forge_forget          — soft-delete with reason
```

### Key Insight

A decision about auth architecture is LINKED to the actual auth code symbols in the SAME graph. Query: "show me all decisions that affect the payment module" returns both architectural decisions AND the code paths they impact.

## Implementation Plan

### Phase 1: Fork and Extend Axon
- Fork github.com/harshkedia177/axon
- Add memory node types (Decision, Pattern, Lesson, Preference, Session) to the schema
- Add memory edge types (SUPERSEDES, MOTIVATED_BY, AFFECTS, etc.)
- Extend the MCP server with forge_remember, forge_recall, forge_link, etc.
- Keep all existing Axon code intelligence tools

### Phase 2: Replace codebase-memory-mcp
- Remove the bundled codebase-memory-mcp binary from Forge
- Point forge-graph MCP server to the extended Axon
- Update all skill/agent MCP tool references
- Migrate manage_adr functionality into Decision nodes

### Phase 3: Active Memory Capture
- SessionStart hook: load relevant decisions/patterns into context
- PostToolUse hook: detect architecture-impacting edits, auto-link to decisions
- SessionEnd hook: extract session decisions, update temporal links
- TaskCompleted hook: store evaluator scores as Pattern confidence data

### Phase 4: Self-Evolution (OpenSpace Integration)
- Integrate OpenSpace's skill evolution patterns
- Use the unified graph to track skill versions and evolution lineage
- Evaluator scores feed back into pattern confidence
- Propose skill mutations with evidence from the graph
- Gate mechanism: test mutations against historical data before applying

## What Gets Dropped

- codebase-memory-mcp (replaced by Axon fork)
- MEMORY.md for structured decisions (replaced by forge_remember)
- STATE.md for session tracking (replaced by Session nodes in graph)
- Flat file ADRs (replaced by Decision nodes with AFFECTS edges to code)

## Success Criteria

1. One `forge-graph` MCP server handles ALL graph queries (code + memory)
2. "Show me all decisions that affect the auth module" returns linked results
3. "What patterns have we learned about error handling?" traverses the graph
4. "How has our API design evolved over the last 5 sessions?" shows temporal edges
5. Memory persists across sessions without flat files
6. New sessions start with relevant context auto-loaded from the graph
