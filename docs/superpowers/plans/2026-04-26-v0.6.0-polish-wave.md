## P3-3.7 — Drift-fixture tests (6 commits, ~2-3 hours)

**Source: bench audit GAP-6.** No "plant a regression, prove the bench
catches it" tests. Each bench infrastructure-checks itself but never
adversarially tests its own sensitivity.

**Pattern per bench:** add a `#[cfg(test)] mod drift_fixtures` to each
`forge_*.rs` containing one or more tests of the form:

```rust
#[test]
fn d1_catches_planted_cross_project_leak() {
    let mut state = setup_clean_state_with_seeded_corpus();

    // PLANT THE BUG: directly INSERT a row that violates the property
    state.conn.execute(
        "INSERT INTO memory (id, project, ...) VALUES ('planted_leak', 'beta', ...)",
        [],
    ).unwrap();
    // ... but tag it so D1's foreign-token detector catches it.

    let dim = dim_1_cross_project_precision(&mut state, &corpus);
    assert!(
        dim.score < dim.min,
        "D1 must catch planted cross-project leak (got {} ≥ min {})",
        dim.score, dim.min
    );
}
```

### P3-3.7 close (2026-04-26) — partial coverage with documented rationale

**Landed (4 drift-fixture tests across 2 benches):**
- W17 (forge-isolation, commit `daa76ad`): 2 tests (D1 cross-project leak, D6 compile-context leak).
- W14 (forge-consolidation, commit `cb05c01`): 2 tests (D1 dedup miss, D1 signal-preservation failure).

**Deferred to v0.6.1+ (W15 + W16 + W18) — three benches whose key
dims aren't drift-testable via direct INSERT planting without adding
production-code injection plumbing:**
- W15 forge-context: D2 guardrail + D4 layer-recall both run
  context-assembly through compile_dynamic_suffix_with_inj which
  materializes from current state — planted state changes get
  overwritten by the next assembly.
- W16 forge-identity: D1 facet-persistence + D2 disposition-drift
  both score against state the daemon recomputes from kpi_events;
  planted memory-table rows don't propagate into the score path.
- W18 forge-coordination: D1 inbox-precision is bound to
  `list_messages` SQL correctness (the WHERE clause filters by
  `to_session=?1` so direct INSERTs with foreign to_session never
  appear in the bench's view); D4 authorization runs the production
  `respond_to_message` code path so a "bypass" requires injecting a
  buggy variant into prod code.

**Forge-evaluation finding (recorded for P3-3.8 dogfood):** the
drift-fixture pattern is high-value but only applies cleanly to
benches whose dims read raw database state. Benches that compose
through production code paths (`list_messages`, `compile_context`,
`respond_to_message`, `kpi_events`-recompute) need a different
sensitivity-testing pattern — likely a `_with_inj`-wrapper variant
of the dim function plus an injected-buggy-callable in the test.
That refactor is out of scope for v0.6.0; tracked for v0.6.1+.
