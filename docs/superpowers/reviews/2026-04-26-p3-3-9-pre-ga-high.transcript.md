## 0. Posture

The 3 fixes are real, end-to-end-verifiable improvements over the broken pre-W20 state. None of them are correctness BLOCKERs in the sense of "merge will produce incorrect on-disk results immediately." But two HIGH issues land:

1. **W22's spawn_blocking task is fire-and-forget with no panic boundary, no shutdown coordination, no concurrency guard** — a real future-DoS or split-brain risk under SIGTERM-mid-pass or panic-mid-pass.
2. **W21 silently descopes the `respond` half of F11** that the plan-doc explicitly committed to, leaving CLI-side reply round-trips inaccessible.

The remaining findings are MEDIUMs and LOWs that I want addressed before opening P3-3.10, but none block the fix wave from being merged-as-is if the user accepts them as deferred-with-known-shortcomings.

## 1. W20 — LD_LIBRARY_PATH forwarding (commit `54aeecd`)

### What changed

The old code inlined a 6-key array right inside the env-forwarding loop. The fix extracts it to `const FORWARDED_ENV_KEYS: &[&str]` at module scope, adds 3 dynamic-loader keys (`LD_LIBRARY_PATH`, `DYLD_LIBRARY_PATH`, `DYLD_FALLBACK_LIBRARY_PATH`), and adds 2 unit tests asserting the const contents.

```rust
const FORWARDED_ENV_KEYS: &[&str] = &[
    "FORGE_PROJECT", "FORGE_PROJECT_DIR", "FORGE_DB", "FORGE_SOCKET",
    "HOME", "PATH",
    "LD_LIBRARY_PATH", "DYLD_LIBRARY_PATH", "DYLD_FALLBACK_LIBRARY_PATH",
];
```

### Things I challenged

**Q1: What about `LD_PRELOAD`, `LD_AUDIT`, `LD_BIND_NOW`?**

These are NOT in the allowlist. After thinking about each:

- `LD_PRELOAD` — security: forwarding it would allow an attacker who controls the parent shell's env to inject an arbitrary shared library into the daemon. Correct to drop. *But the rationale is undocumented.*
- `LD_AUDIT` — same attack surface as `LD_PRELOAD` (audit hooks run with daemon privileges). Correct to drop.
- `LD_BIND_NOW` — controls lazy vs. eager symbol resolution. Not load-blocking, mostly a debug aid. Reasonable to drop.
- `LD_DEBUG` — diagnostics. Could be useful for support; reasonable to drop in production.
- macOS analogs: `DYLD_INSERT_LIBRARIES` (LD_PRELOAD-equivalent — must NOT be forwarded), `DYLD_FRAMEWORK_PATH`, `DYLD_VERSIONED_FRAMEWORK_PATH`. The first is correctly dropped; the latter two could matter for framework-based ORT builds but the bundled tarball uses dylib not framework, so OK.

Verdict on Q1: forwarding is correctly conservative. Filed `LOW-2` because the security rationale is reconstructable but undocumented.

**Q2: What about Windows?**

Verified `.github/workflows/ci.yml` matrix is `[ubuntu-latest, macos-latest]` — no `windows-latest`. `scripts/setup-dev-env.sh` only handles Linux glibc<2.38. Windows uses `PATH` for DLL search by default; `PATH` is already in the list. So no Windows-specific gap exists today. Filed as `RESOLVED-8`.

**Q3: Are the unit tests meaningful?**

```rust
#[test]
fn forwarded_env_keys_include_dynamic_loader_paths() {
    for required in ["LD_LIBRARY_PATH", "DYLD_LIBRARY_PATH", "DYLD_FALLBACK_LIBRARY_PATH"] {
        assert!(FORWARDED_ENV_KEYS.contains(&required), "...");
    }
}
```

This is a pure constant-array contains-check. It would pass even if the runtime forwarding loop (lines 130-135) were entirely deleted. The tests don't exercise the actual `std::env::var(key) -> envs.push((key, v)) -> Command::envs(envs)` chain. The end-to-end behavior is verified only by the live `/proc/<daemon>/environ` check in the commit message — not regression-locked. Filed `LOW-3`.

A meaningful test would `set_var("LD_LIBRARY_PATH", "/test/path")`, then mock-spawn `bash -c 'echo $LD_LIBRARY_PATH'`, capture stdout, assert it equals `/test/path`. As-is, dropping the runtime forwarding loop would not fail any unit test.

### W20 conclusion

The fix is real and end-to-end-verified. It correctly forwards the dynamic-loader vars on Linux and macOS without opening up `LD_PRELOAD`-class attack surface. The unit tests are tautological shape-checks (LOW-3); the security rationale for the exclusions is undocumented (LOW-2). No HIGH issues.

## 2. W21 — `--from` flag + `FORGE_SESSION_ID` env (commit `6e27eb4`)

### What changed

New helper `commands::system::resolve_from_session(explicit) -> Option<String>` with precedence: `explicit > env > None`. The `Send` clap subcommand gains a `--from <session_id>` arg. `send_message` and `team_send` both route their `from_session` field through the new resolver. 4 unit tests pin the resolution matrix.

### Things I challenged

**Q1: Does `Request::SessionRespond` also need `from_session`?**

The plan-doc (line 67-68) explicitly reads:
> `crates/cli/src/main.rs — add --from <SESSION_ID> flag to the Send subcommand. Same for respond.`

Reality:
1. `Request::SessionRespond` (request.rs:506-510) has fields `{message_id, status, parts}` — no `from_session` field at all.
2. `grep -n "respond" crates/cli/src/main.rs` returns ZERO references to a `respond` subcommand. There is no `forge-next respond` CLI surface today.

So `respond` was descoped. The commit message and HANDOFF don't acknowledge this — they treat W21 as "F11 + F13 closed." From a planner's perspective, F11 is **half-closed**: outbound caller identity is fixed for `send`/`team-send`, but the round-trip reply leg has neither a CLI subcommand nor an identity field. Filed `HIGH-2` — not a correctness BLOCKER (the existing code path works correctly), but a *plan-completeness* HIGH because the plan-doc explicitly committed to it and a future reader will assume both halves shipped.

**Q2: Is the `eprintln!` warning actually "one-time"?**

The doc-comment at line 851 reads:
> Emits a one-time stderr warning when neither source is present...

The implementation (line 861) is a bare `eprintln!`. No `std::sync::Once`, no static `AtomicBool`. In practice this is fine for `forge-next` (one-shot CLI process), but the comment phrasing is a regression trap if a future feature loops `send_message` from a wrapper script or moves to a long-lived REPL. Filed `MED-2`.

**Q3: Does `--from ""` correctly fall through to env?**

```rust
if let Some(v) = explicit.filter(|s| !s.is_empty()) {
    return Some(v);
}
```

Yes — `Some("")` is filtered out, so `--from ""` falls through to `FORGE_SESSION_ID`. Test `empty_flag_falls_through_to_env` pins this. RESOLVED-2.

**Q4: What about whitespace? `--from "  "` (whitespace-only)?**

The filter is `s.is_empty()`, not `s.trim().is_empty()`. So `--from " "` becomes `from_session=" "` — valid identity, daemon will record it as such. Borderline UX but consistent with not silently mutating user input. Not filed (acceptable).

**Q5: Did W21 change team-send semantics?**

Pre-W21 `team_send` had `from_session: from` (raw passthrough — `--from x` worked, but `FORGE_SESSION_ID=x` did NOT). Post-W21 routes through `resolve_from_session(from)` — both work. This is a *strengthening* (env now consulted), backwards-compatible (explicit --from still wins). RESOLVED-3.

**Q6: Are the env-var tests safe under parallel cargo test?**

```rust
static ENV_LOCK: Mutex<()> = Mutex::new(());

#[test]
fn explicit_from_flag_wins_over_env() {
    let _guard = ENV_LOCK.lock().unwrap();
    std::env::set_var("FORGE_SESSION_ID", "ses-from-env");
    let resolved = resolve_from_session(Some("ses-from-flag".to_string()));
    std::env::remove_var("FORGE_SESSION_ID");
    ...
}
```

The `Mutex<()>` serialises the four tests against each other within the binary. **But**:
- If `resolve_from_session` panics (or any future change adds an `assert!`), the panic propagates with the lock still held AND the env var still set.
- Result: (a) sibling tests in the same process see the leaked `FORGE_SESSION_ID`; (b) the next test that calls `lock().unwrap()` fails with `PoisonError`, masking the original panic.
- Standard Rust pattern is `temp_env::with_var` or a `Drop`-based RAII guard that calls `remove_var` regardless of unwind.

Filed `LOW-1`. Today the tests pass because `resolve_from_session` is too simple to panic, but the pattern is fragile.

**Q7: Could `FORGE_SESSION_ID` leak from sibling tests in the SAME crate but DIFFERENT modules that don't use this lock?**

Searched: `grep -rn "FORGE_SESSION_ID" crates/` — only the four W21 tests reference it. No sibling-test leak vector today. Acceptable.

### W21 conclusion

The fix correctly closes F11+F13 for the `send`/`team-send` outbound path, with a sensible precedence chain and reasonable empty-string handling. The descoping of `respond` is the most actionable finding — either implement the missing CLI subcommand+protocol field in this wave, or explicitly document the descoping in HANDOFF. The "one-time" comment vs. unguarded `eprintln!` is a doc/code drift. The env-mutex test pattern is fragile but tests-only.

## 3. W22 — force-index async (commit `611169b`)

### What changed

`Request::ForceIndex` is now intercepted in `WriterActor::run`'s match arms (both `WriteCommand::Raw` and `WriteCommand::Audited`). The interceptor calls `process_force_index_async(path)` which:
1. Synchronously canonicalizes the path (so callers still get immediate "not a directory" errors).
2. Spawns a `tokio::task::spawn_blocking` worker that opens its own SQLite connection and runs the actual indexing.
3. Returns `IndexComplete { 0, 0 }` immediately so the writer queue moves on.

The `db_path: String` field is added to `DaemonState` so the spawn_blocking closure can open a fresh connection.

CLI side: `force_index` reads `(0, 0)` as "background started" and prints a hint message.

### Things I challenged

**Q1: Does dropping the JoinHandle swallow panics silently?**

Yes. Tokio's `spawn_blocking` returns a `JoinHandle` that, if dropped without await:
- The blocking thread continues running to completion (Tokio doesn't abort blocking threads when the handle drops).
- If the thread panics, the panic is captured in `JoinError` but never observed because nobody awaits.
- Default Rust panic hook prints to stderr (the daemon log captures this), but the daemon has no programmatic signal that indexing failed.

Compare `kpi_reaper::run_reap_blocking`:
```rust
let join = tokio::task::spawn_blocking(move || -> rusqlite::Result<usize> { ... }).await;
match join {
    Ok(Ok(n)) => tracing::info!(...),
    Ok(Err(e)) => tracing::error!(...),
    Err(e) => tracing::error!(target: "...", panic = %e, "reaper task panicked"),
}
```

The W22 code doesn't follow this precedent. Filed as part of `HIGH-1`.

**Q2: Does the daemon's SIGTERM handler know about the spawn_blocking task?**

Verified from `crates/daemon/src/main.rs` lines 378-440: the SIGTERM handler routes through a `watch::channel(false)` and is subscribed by HTTP/gRPC servers. The writer-actor's spawn_blocking task does NOT subscribe to the shutdown channel. On SIGTERM:
- The runtime stops accepting new work.
- The spawn_blocking task continues running to completion OR is aborted when the runtime drops.
- The actual behavior depends on Tokio runtime drop semantics — the runtime's `Drop` impl calls `shutdown_background()` which **does NOT wait for blocking tasks**. They are abandoned mid-execution.

Result: indexer can be killed mid-INSERT. SQLite WAL is robust against this at the *atomic-commit* level (no torn pages), but the indexer pass writes many transactions (one per file batch + one per cluster). A killed pass leaves a half-indexed tree until the next force-index.

The previous synchronous handler held the writer mutex until completion — SIGTERM-during-indexing blocked the daemon's SIGTERM-handler from sending shutdown_tx until the pass finished. That had its own DoS-on-shutdown problem (which is exactly why W22 exists). The fix trades writer-DoS for split-brain-on-shutdown. Both are acceptable in different deployments, but the choice is undocumented. Filed as part of `HIGH-1`.

**Q3: Is the path validation TOCTOU-safe?**

Look at the data flow:
```rust
fn process_force_index_async(&self, path: Option<String>) -> Response {
    if let Some(ref dir) = path {
        match std::fs::canonicalize(dir) {        // <-- 1st canonicalize (validation)
            Ok(canonical) if canonical.is_dir() => { /* OK */ }
            ...
        }
    }
    let path_for_task = path;                      // <-- raw `dir` passed in
    tokio::task::spawn_blocking(move || {
        run_force_index_in_task(&db_path, path_for_task);
    });
}

fn run_force_index_in_task(db_path: &str, path: Option<String>) {
    if let Some(dir) = path {
        let canonical = std::fs::canonicalize(&dir)  // <-- 2nd canonicalize
            .map(|p| p.to_string_lossy().to_string())
            .unwrap_or(dir);                          // <-- silently falls back to raw
        ...
    }
}
```

Between the 1st and 2nd canonicalize, an attacker (or a concurrent `mv` in the user's shell) can:
- Rename the directory: 1st canonicalize sees `/foo/bar`, 2nd sees `/foo/bar` not-found, falls through `unwrap_or(dir)` to walk the literal `dir` (could be a different relative path).
- Replace with a symlink: 2nd canonicalize follows the new symlink to a target outside the original tree.
- Delete: 2nd canonicalize errors, falls through to walk `dir` literally — `index_directory_sync` may walk an empty/missing tree and silently produce 0 results.

Since `force-index` is admin-RBAC-gated (handler.rs:4674), the impact is bounded — the operator is the attacker, and they can already destroy the workspace. But this introduces a NEW window that didn't exist in the synchronous handler (which canonicalised exactly once). Filed `MED-1`.

Mitigation: pass the canonical path from the synchronous validation into the closure (avoid recanonicalize entirely). Or hold an `O_PATH` fd on Linux to pin the inode.

**Q4: Is `init_sqlite_vec()` correctly idempotent?**

Verified `db/vec.rs:11-20`: uses `std::sync::Once::call_once`. Safe to call from any thread. RESOLVED-6.

**Q5: Is WAL inherited by the new connection?**

WAL mode is set by `PRAGMA journal_mode=WAL` and persists in the database header (per https://sqlite.org/wal.html §3). Subsequent connections to the same file automatically use WAL without needing the PRAGMA. So the explicit PRAGMA in `run_force_index_in_task` is redundant on a daemon-initialised DB — but harmless. RESOLVED-9.

**Q6: Is there deadlock risk with concurrent force-index calls?**

If two `force-index` requests arrive within seconds:
- Both spawn separate `spawn_blocking` tasks.
- Both open separate connections to the same DB.
- Both call `index_directory_sync` which writes to `code_file`, `code_symbol`, `import_edge`, `cluster` tables.
- SQLite WAL serialises writes at the *transaction* level via the `-wal-index` lock.
- Each individual transaction (one per file batch) succeeds in turn. No deadlock.

So no deadlock — but two redundant indexing passes burn CPU and write the same rows twice (idempotent with `INSERT OR REPLACE` patterns). Cosmetic for one user; a real DoS vector if a script accidentally loops `force-index`. Filed as part of `HIGH-1` (concurrency guard recommended).

**Q7: Will the CLI's `(0, 0) → "background started"` heuristic produce false positives?**

Three failure modes:
1. **Empty project**: a brand-new directory with zero indexable files legitimately returns `(0, 0)`. User sees "Indexer dispatched in background. Watch ~/.forge/daemon.log" — *misleading* (nothing dispatched, nothing in the log).
2. **Spawn task panic**: indexer task panics mid-pass, but the CLI already returned `(0, 0)`. User waits for completion that never happens, sees no error.
3. **Pre-W22 daemon synchronous-empty case**: an old daemon truly returning `(0, 0)` from a sync run also gets the new misleading message.

Filed `MED-3`. The cleanest fix is a new protocol variant (`ResponseData::IndexStarted { task_id }`) but that's a breaking change. A tactical fix is to add a `dispatched_async: bool` field, but that requires a harness-sync update.

**Q8: Is the protocol change a hidden harness-sync drift?**

Verified `bash scripts/check-harness-sync.sh` returns clean. The wire-level shape of `ResponseData::IndexComplete` is unchanged (`files_indexed: usize, symbols_indexed: usize`). What changed is the *meaning* of the values. No JSON-method or CLI-subcommand list updated. Pre-W22 scripts that pipe `forge-next force-index --json | jq '.symbols_indexed > 0'` will silently break their semantics check (no error, just wrong answer). I searched for such consumers in `tests/`, `scripts/`, `hooks/` — none exist today. Filed `LOW-5` as a footgun rather than an immediate break.

**Q9: Why `busy_timeout=10000` instead of the canonical `5000`?**

Looking at precedents:
- `handler.rs` `new`/`new_writer`/`new_reader`: all use `busy_timeout=5000`.
- `kpi_reaper.rs:246`: uses `busy_timeout=5000`.
- `writer.rs:382` (W22): uses `busy_timeout=10000`.

Inconsistent. The doubled value isn't catastrophic (just wait twice as long for a contended write before erroring), but the rationale is undocumented. Filed `MED-4`.

**Q10: The tracing-format `?path_for_task`?**

```rust
tracing::info!(target: "...", ?path_for_task, "force-index dispatched...");
```

Formats `path_for_task` via `Debug` — produces ugly log lines like `path_for_task=Some("/foo/bar")` for set values and `path_for_task=None` for unset. Cosmetic; filed `NIT-1`.

**Q11: Live verification timings — reproducible?**

Commit message claims:
- `force-index` 0.008s (was 30s timeout)
- `health` (concurrent) 0.007s
- `sessions` (concurrent) 0.006s
- `send` (concurrent) 9.4s (was 30s)

I can't reproduce these directly without rebuilding and running the daemon, but the architecture analysis supports the claim:
- `force-index` returns immediately after dispatch — order of microseconds for the actor match arm + spawn — `0.008s` is plausible (network round-trip dominated).
- `health`/`sessions` are reads — they bypass the writer-actor entirely, never blocked by force-index. Plausible.
- `send` is a write — queues behind the writer-actor's CURRENT command, which is now the cheap `IndexComplete{0,0}` return rather than a 30s indexing pass. The 9.4s residual latency likely comes from inter-connection WAL contention as the spawn_blocking task hammers writes; commit message acknowledges this. Plausible.

The numbers are internally consistent. RESOLVED implicitly (not filed).

### W22 conclusion

The architectural fix is correct and the live verification is plausible. But the spawn_blocking task is a fire-and-forget pattern that doesn't follow the kpi_reaper precedent: no panic boundary, no shutdown coordination, no concurrency guard. These are HIGH because they create new failure modes (silent panic, split-brain SIGTERM, redundant concurrent passes) — none of which are immediately observable from the smoke test, but all of which will surface under real production wear. The TOCTOU on path canonicalisation, the undocumented busy_timeout choice, the misleading CLI message on legitimately-empty projects, and the protocol-meaning shift without a wire-shape change are MEDIUMs that I want addressed before P3-3.10.
