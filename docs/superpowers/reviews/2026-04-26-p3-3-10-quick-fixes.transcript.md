### `b965d0b` — W25 daemon-spawn polish

**F1 stale-version.** Compares `cli_version` (compile-time
`CARGO_PKG_VERSION`) vs daemon-reported `version` as a plain string equality.
Limitations:

1. (LOW) **Same Cargo.toml version, different build commits.** If the user
   `git pull`s master and rebuilds without bumping `CARGO_PKG_VERSION`,
   the CLI and daemon will both report `0.6.0-rc.3` — but one is from a
   different commit. The `git_sha` field IS already in the `Doctor`
   response (rendered in `sha_str`); the F1 fix could opportunistically
   compare `git_sha` too and flag a sha mismatch with the same `[stale
   daemon...]` decoration. The current fix is correct for its stated
   condition (version bump) but misses the more common dev workflow
   (in-progress branch with stable Cargo version).

2. (NIT) The decoration is a long English sentence appended to the
   `Version:` line: `0.6.0-rc.3 (bd1bac6) [stale daemon, CLI is v0.6.0-rc.4
   — run \`forge-next restart\`]` — this can blow the 80-char terminal
   width. Consider a separate `[STALE]` health-check entry instead.

**F2 hooks-gate.** Verified the gate logic via reading `handler.rs:1755-1789`:

- `hook_paths` checks for `~/.claude/plugins/forge/hooks/hooks.json` OR
  `$CLAUDE_PLUGIN_ROOT/hooks/hooks.json`.
- `plugin_install_dirs` checks for `~/.claude/plugins/forge/` OR
  `$CLAUDE_PLUGIN_ROOT`.
- `match` arms: `Some(p) → ok+count`, `None if has_plugin_install → warn`,

The gate is structurally correct: a user with the plugin install root but

3. (LOW) **Symlink subtlety.** `is_dir()` follows symlinks. If

   detect a broken-symlink plugin install.

**F3 timeout.** 100 × 100ms = 10s is reasonable; ONNX 5–7s is documented in

### `eb55a2d` — W26 team primitives

**F6 idempotent run_team.** Verified the upsert logic in `teams.rs:567-578`.
The `team_pre_existed` flag correctly gates the rollback `DELETE FROM team`,
so a manually-created team isn't deleted when a downstream spawn fails on
the second run.

Two test-passing concerns:

1. (LOW) **Schema lacks ON DELETE CASCADE.** `team_member` has
   `PRIMARY KEY (team_id, user_id)` but **no foreign key on team_id
   referencing team(id)** (verified `db/schema.rs:610-616`). Combined with
   the daemon not setting `PRAGMA foreign_keys = ON` (verified — no
   matches), this means: on the rollback path where W26 deletes the team
   row (when `!team_pre_existed`), any `team_member` rows already inserted
   in this run for **earlier successful** spawns are NOT deleted, leaving
   orphaned `team_member` rows pointing at a non-existent team_id. This is
   pre-existing behavior (not regressed by W26), but W26's idempotent
   upsert makes the orphaned-rows path more reachable: subsequent runs
   that reuse the team see `member_count` inflated by the orphans
   (since the orphan team_id won't match the new run's team_id, no
   collision; but a later schema migration that adds the FK would fail).

2. (MEDIUM) **No team_member rollback on per-agent failure.** When
   `spawn_agent` fails partway through a run with a pre-existing team, the
   rollback only retires/ends sessions and skips the team-delete
   (correctly, since pre-existed). But it does NOT explicitly delete the
   `team_member` rows that the prior successful spawns inserted into the
   pre-existing team. So a partial-failure run leaves the pre-existing
   team with member rows pointing at retired sessions. The next
   `team members` listing will show those retired sessions because
   `list_team_members` joins on `team_member.session_id` without filtering
   by session status (verified). This is a real leak, not just cosmetic.

**F7 stop wording.** The annotation `(team had no spawned agents)` only
fires on `agents_retired == 0`. Looked up `stop_team` in
`teams.rs` — verified that the count only includes successful retires;
errored retires are silently dropped. Acceptable for now, but the CLI's
unconditional success message hides partial-retire failures.

3. (LOW) **stop_team partial-failure is invisible.** If `stop_team`
   succeeds at retiring 2 of 3 agents (3rd errors), the CLI reports
   `2 agent(s) retired` with no annotation. Combined with the new F7
   wording on `0`, the user sees `0 agent(s) retired (team had no spawned
   agents)` even when, in reality, retire returned errors for all 3
   spawned agents. Distinguishing "team genuinely empty" from "all retires
   failed" requires the daemon to expose an `errored_count` alongside
   `agents_retired`.

**F8 --project propagation.** Verified the test
`test_run_team_propagates_project` correctly checks `session.project` (not
`team.project`) — `spawn_agent` calls `register_session(conn, ..., project,
...)` which writes to the `session.project` column. Verified `session`
schema has `project TEXT` column.

4. (LOW) **No project-string validation.** A user passing
   `--project foo` for a project that doesn't exist in any reality table

   cross-check `project_path` against `reality.project_path`. If the
   intent is "scope to a known project", a typo silently scopes work to

   was already permissive), but the new `--project` surface makes the
   typo path easier to hit.

**F9 role display.** The CLI now reads `m.get("role")` and
`m.get("agent_status")`, matching the daemon's output keys. Verified the
daemon emits these in `list_team_members` (in `teams.rs`).

5. (LOW) **No JSON-shape contract test.** The F9 fix relies on the daemon

### `85712a8` — W27 message-read

**Variant census.** Verified by running the contract test:
`unit_count(20) + param_count(104) = 124 expected_total`, all match;
`cargo test -p forge-core --lib test_variant_count_completeness` passes.

**`read_message_by_id_or_prefix`.** Two real concerns:

6. (HIGH) **Privilege/scope regression vs `SessionMessages`.** The prior
   code path for `forge-next message-read` was indirect: it called
   `Request::SessionMessages { session_id: <empty> }` which the daemon
   scopes by `to_session = ?1` (verified `sessions.rs:491`). Even though
   the empty session_id matched no rows, the architectural constraint was
   "you can only see messages addressed to your session". The new
   `read_message_by_id_or_prefix` does NOT scope by `to_session` — any
   caller can read ANY message in the database by its ID (or 8-char
   prefix from the public listing). The daemon is single-tenant so this
   doesn't cross a hard auth boundary, but defense-in-depth is weakened:
   a session that previously could only `messages` its own inbox can now
   directly read messages addressed to other sessions. **Recommend:** add
   an optional `caller_session: Option<String>` to
   `Request::SessionMessageRead` and, when set, scope the SQL with
   `(to_session = ?2 OR from_session = ?2)`. Default open is acceptable
   because today's daemon has no auth, but the field gives future RBAC a
   hook.

7. (LOW) **LIKE wildcard literalness.** `WHERE id LIKE ?1 || '%'` —
   rusqlite parameter-binds `?1` as a literal string, but the LIKE
   operator itself interprets `%` and `_` IN the bound string as
   wildcards. ULIDs use Crockford base32 (`0-9A-HJKMNP-TV-Z`) which
   excludes both `%` and `_`, so a real ULID prefix is safe. But:
   `forge-next message-read "%"` matches every message; if exactly one
   row exists (e.g., a dev-fresh DB), it's returned. Worth either
   ESCAPE-ing the `%`/`_` chars before binding (use
   `LIKE ?1 || '%' ESCAPE '\\'` and replace `%`/`_` with escaped forms)
   or rejecting non-ULID-charset input at the helper boundary.

8. (LOW) **No `.trim()` on the CLI input.** If the user copy-pastes a
   ULID with a trailing space (common terminal-paste hazard), the daemon
   does `id = "01ABC...Z "` — exact match fails (trailing space), then
   prefix match also fails (the trailing space is included in the LIKE
   pattern, no row's id ends with a space). Result: `message not
   found: 01ABC...Z ` (with the offending space invisible). Recommend a
   `.trim()` in the CLI before sending, or in the daemon helper.

9. (LOW) **Ambiguity error wrapped via InvalidParameterName.** The
   helper returns `Err(rusqlite::Error::InvalidParameterName(...))` on
   ambiguity, which the handler wraps as `format!("session_message_read
   failed: {e}")`. The user sees `error: session_message_read failed:
   ambiguous message ID prefix '01ABC' — type more characters (matches
   at least 2 rows)`. The "type more characters" hint IS preserved
   through the formatting, but the leading `session_message_read failed:`
   is noise. Better: split ambiguity from real errors via a dedicated
   error variant (`AmbiguousPrefix`) and let the handler format it as
   `Response::Error { message: "ambiguous prefix '...'..." }`.

10. (NIT) **8-char display prefix may already be ambiguous.** The
    `messages` listing in `cli/src/commands/system.rs:969` truncates to
    8 chars. ULIDs are time-sortable; 8 chars represents ~40 bits of the
    timestamp prefix. For projects with thousands of messages within the
    same minute, 8 chars will collide. The new prefix-resolution
    correctly returns `ambiguous` in that case, but the `messages`
    listing should display 12+ chars to keep the copy-paste workflow
    deterministic.

**Test coverage.** No new daemon-side unit test for
`read_message_by_id_or_prefix`. The contract test covers serialisation
roundtrip but not the helper's behavior (exact match, prefix match, 0
rows, ≥2 rows). Recommend adding 4 tests in `sessions.rs::tests`.

11. (LOW) **No unit test for the resolver helper.** The function has
    four behavioral branches (exact-hit, prefix-1, prefix-0, prefix-≥2)
    but only the live verification in the commit message exercises any
    of them. A regression that flips the prefix order to "prefix first,
    exact second" would be caught only at integration time.
