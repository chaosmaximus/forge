## Key findings (chronological narrative)

### Step 1 ‚Äî fastembed pin sanity

`fastembed = "=5.11.0"` is a strict equality requirement (Cargo's `=` operator). Lock pins `ort 2.0.0-rc.11`. `cargo update` cannot drift because (a) the pin forbids any other version of fastembed, and (b) ort is a transitive whose manifest is dictated by fastembed-5.11.0. `Cargo.lock` deletion + blind `cargo update --workspace` is the only escape ‚Äî there's no `deny.toml` rule pinning ort directly. The Cargo.toml comment explicitly names the unpin condition (ONNX RT 1.24+ in `.tools/`) and lists the three files to bump (`scripts/setup-dev-env.sh`, `.cargo/config.toml`, `scripts/with-ort.sh`). Adequate. **LOW**: setup-dev-env.sh's ORT_VERSION="1.23.0" line has no cross-reference back to fastembed ‚Äî an operator bumping ONNX RT might forget to unpin. Cosmetic.

### Step 2 ‚Äî the migration SQL

```sql
UPDATE code_file
SET project = SUBSTR(project, LENGTH(project) - INSTR(REVERSE(project), '/') + 2)
WHERE project LIKE '/%' AND INSTR(REVERSE(project), '/') > 0
```

Walked the math for `/foo`, `/foo/bar`, `/foo/`, `/`:

* `/foo` ‚Üí SUBSTR(_, 4-4+2=2) ‚Üí `foo` ‚úì
* `/foo/bar` ‚Üí SUBSTR(_, 8-4+2=6) ‚Üí `bar` ‚úì
* `/foo/` ‚Üí SUBSTR(_, 5-1+2=6) ‚Üí `''` (empty) ‚Äî the second UPDATE catches this and turns it into `_global_`. **A real project tagged `/foo/` (trailing slash) loses its name to the sentinel.** Production indexer paths from `tempfile::tempdir()` etc. don't have trailing slashes, but it's an edge case worth flagging (LOW).
* `/` ‚Üí SUBSTR(_, 1-1+2=2) ‚Üí `''` ‚Äî sentinel (acceptable).

**Bigger gap:** W29 and W30 each shipped a dedicated migration regression test (`p3_3_11_w29_project_sentinel_backfill`, `p3_3_11_w30_identity_project_sentinel_backfill`) that builds a pre-migration table shape, runs `create_schema`, and asserts post-migration values. **W1.2 c1 ships no equivalent test.** The PATH‚Üíbasename SUBSTR/REVERSE/INSTR backfill is unverified by any cargo test. This is an explicit deviation from the W29/W30 pattern in the same migration block. **MED**.

### Step 4 ‚Äî the rename-vs-path mismatch downstream of `f.project`

After c1's migration, `code_file.project` stores a NAME (e.g. `forge`). But two production sites still treat it as a PATH:

* `crates/daemon/src/server/handler.rs:4918-4920` ‚Äî `Request::ForceIndex { path: None }` re-process branch. Builds a HashSet of `f.project.clone()` from `list_code_files`, then calls `run_clustering(&state.conn, project_dir)` for each. After migration `project_dir` = `forge` not `/mnt/.../forge`.
* `crates/daemon/src/server/writer.rs:402-405` ‚Äî same pattern on the writer-queue path.

`run_clustering` calls `get_reality_by_path(conn, project_dir, "default")` which matches `reality.project_path = ?1` exactly. With a NAME input, that lookup returns None and the early-return path skips clustering. **Net effect: every no-path force-index post-W1.2-migration silently skips clustering entirely.** Clustering is the source of `cluster_files`/`cluster_name` in BlastRadius output. **HIGH**. The indexer's *internal* call sites at lines 836/941 still pass the original `project_dir` parameter (the PATH), so first-time index continues to cluster correctly. The break is only on re-process.

Same shape applies to `auto_detect_conventions` if it were called from these branches ‚Äî it's not, but the function reads the conventions-file relative to `project_dir`, so a NAME input would silently no-op there too.

This is the SQL-alias-shadowing memory pattern manifesting at the application layer: same column name, semantically different data on different code-paths.

### Step 5 ‚Äî pre-migration data carryover

The c3 commit message claims `forge|188` only on a fresh DB (post-wipe). Verification was on a wiped DB. The migration in c1 *renames* `mnt`-PATH-tagged rows to `mnt`-NAME-tagged rows. It does NOT delete code_file rows for foreign projects, does NOT clean up code_symbol rows for those file_paths, does NOT clean up edge rows. So a previously polluted production DB that upgrades through the migration without a wipe will have:

* code_file rows tagged `project = 'mnt'` (or `home`, `usr`, etc., basenames of whatever shallow root the pre-c3 fallback grounded at).
* `forge-next find-symbol foo --project mnt` returns 10,005 symbols from the foreign user's $HOME.

The HANDOFF documents the wipe (`forge.db.pre-W1.2-wipe-20260426-145000.bak (219 MB pre-wipe baseline)`) as a backup, but there's no operator-facing migration note. **HIGH**: silent data carryover for any user who upgrades without wiping. The fix is either a one-shot `DELETE FROM code_file WHERE project IN ('mnt','home','usr','var','tmp')` companion migration, OR a documented operator wipe step in the HANDOFF / changelog.

### Step 11 ‚Äî index efficacy on the JOIN

The new `idx_code_file_project` is a single-column index on `code_file(project)`. The JOIN SQL is:

```sql
SELECT s.id, s.name, s.kind, s.file_path, s.line_start
FROM code_symbol s
JOIN code_file f ON s.file_path = f.path
WHERE s.name LIKE ?1 AND f.project = ?2 LIMIT ?3
```

SQLite query planner has multiple strategies: (a) scan `code_symbol` filtered by `idx_code_symbol_name`, then per-row indexed lookup on `code_file.path` (via `idx_code_file_path` UNIQUE), then check `f.project` post-filter; (b) start from `code_file` with `idx_code_file_project = ?2`, then JOIN `code_symbol` via `idx_code_symbol_file`. (b) is more selective when the project has a small fraction of files (e.g. forge=188 / 10193 ‚âà 1.8%); (a) is better when the LIKE pattern is highly selective. SQLite chooses based on stats; without ANALYZE the choice is fragile. A composite `(project, path)` covering index would be more robust. Performance ‚Äî not correctness ‚Äî concern. **LOW**.

### Step 12 ‚Äî depth threshold of 4

`/mnt/colab-disk/DurgaSaiK/forge` has 4 slashes ‚Äî admitted. `/home/user/projects/foo` (4 slashes) admitted. `/mnt/foo/bar/baz` (4 slashes) admitted ‚Äî could be a system mount, but the `is_dir()` check still applies in production so a non-existent path here returns None. The threshold is empirically sized for the dev's host layout (`/mnt/colab-disk/...`). On a server where projects live at `/srv/foo` (2 slashes), the depth-floor would reject every legitimate project. Not a current issue, but the threshold is host-shape coupled. The decode-fallback memory itself recommends a marker-file fallback (Cargo.toml/.git) as the better long-term play; no tracking issue exists for that follow-up. **LOW**.

### Step 13 ‚Äî `derive_project_name` reality-scope hardcoding

`derive_project_name` calls `get_reality_by_path(conn, ..., "default")` ‚Äî hardcoded scope. Multi-org deployments use non-default scopes; the helper would silently fall back to basename for any non-default-scope reality. Since the function is dead code today, this is dormant. If a future caller wires it in, this will surface as a multi-tenant correctness bug. **LOW** (preventive flag).
