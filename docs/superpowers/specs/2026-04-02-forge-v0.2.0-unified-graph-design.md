# Forge v0.2.0 Design Spec: Unified Knowledge Graph + HUD

**Date:** 2026-04-02
**Author:** chaosmaximus
**Status:** Approved

---

## 1. Vision

Replace Forge's fragmented memory architecture with ONE unified LadybugDB-powered knowledge graph that holds code structure, architectural decisions, learned patterns, lessons, session history, and skill evolution data in a single queryable graph. Add a real-time HUD for observability and enforce deterministic-first execution discipline with full token tracking.

**Design principles:**
- One graph, one engine, everything connected
- Deterministic paths first — LLM only where judgment is required
- Every token tracked, every ratio visible
- User is always the gate — no semantic mutation (decisions, patterns, skills, code) without approval. Operational state (sessions, metrics, HUD) updates automatically.
- Fast and optimized — measured end-to-end SLOs, not slogans
- Trust boundaries explicit — every data flow classified by trust level

**Key decisions:**
- Axon used as a library (`import axoniq`) — not forked
- Shared LadybugDB file — Axon writes code nodes, we write memory nodes
- One `plugin.json` MCP entry replaces `codebase-memory-mcp`
- Python package: `pip install forge-graph` (includes `axoniq` as dependency)
- Binary option: PyInstaller bundle (~80MB) for users without Python
- HUD: Rust, ~2MB static binary, <2ms per poll, uses Claude Code's native statusLine API

## 3. Unified Graph Schema

### 3.1 Code Nodes (from Axon — untouched)

```
File, Folder, Function, Class, Method, Interface, TypeAlias, Enum, Community, Process
```

### 3.2 Code Edges (from Axon — untouched)

```
CONTAINS, DEFINES, CALLS, IMPORTS, EXTENDS, IMPLEMENTS, MEMBER_OF,
STEP_IN_PROCESS, USES_TYPE, EXPORTS, COUPLED_WITH
```

### 3.3 Memory Nodes (new)

| Node | Properties | Purpose |
|------|-----------|---------|
| `Decision` | `id, title, rationale, status (active/superseded/rejected), created_at, updated_at, valid_at, invalid_at, confidence, trust_level (user/agent)` | Architectural choices |
| `Pattern` | `id, name, description, domain, frequency, confidence, generation, created_at, updated_at, valid_at, invalid_at` | Learned recurring approaches |
| `Lesson` | `id, insight, context, severity (info/warning/critical), created_at, updated_at, valid_at, invalid_at` | What went wrong/right |
| `Preference` | `id, key, value, scope (user/project), confidence, created_at, updated_at, valid_at, invalid_at` | User behavioral preferences |
| `Session` | `id, started_at, ended_at, mode (greenfield/feature), project, outcome, summary, total_tokens_input, total_tokens_output, total_llm_calls, total_tool_calls, deterministic_ratio` | Session provenance + token tracking |
| `Skill` | `id, name, version, generation, is_active, content_hash, total_selections, total_applied, total_completions, total_fallbacks, created_at, updated_at` | Evolution tracking |
| `Secret` | `id, type (api_key/token/cert/password/ssh_key/env_var), provider (aws/gcp/azure/github/stripe/generic), file_path, line_number, discovered_at, last_rotated, age_days, risk_level (critical/high/medium/low), status (active/rotated/revoked/expired), fingerprint` | Tracked secrets (NO secret values stored — fingerprint only) |

**Schema fixes (P2):**
- Provenance via edges only (`DECIDED_IN`, `LEARNED_IN`) — no duplicate `session_id`/`source_session` scalar fields.
- `valid_at`/`invalid_at` on ALL memory nodes (not just edges) — unified temporal contract.
- `Skill.metrics_json` replaced with typed counters (`total_selections`, `total_applied`, `total_completions`, `total_fallbacks`) — no opaque blobs in query core.
- `EVOLVED_FROM.diff` stored as a separate file reference, not inline blob.
- `trust_level` on Decision nodes: `user` (explicitly confirmed) vs `agent` (auto-detected). Agent-trust nodes are sanitized before context injection.

### 3.4 Memory Edges (new)

| Edge | From → To | Properties | Purpose |
|------|----------|-----------|---------|
| `SUPERSEDES` | Decision → Decision | `valid_at, invalid_at, reason` | Decision evolution |
| `MOTIVATED_BY` | Decision → Lesson | `strength` | Why a decision was made |
| `FOLLOWS` | Decision → Pattern | `confidence` | Decision implements a pattern |
| `CONTRADICTS` | Decision → Decision | `detected_at, resolved` | Conflicting choices |
| `AFFECTS` | Decision → Function\|Class\|File | `impact (high/med/low)` | Links decisions to code |
| `LEARNED_IN` | Lesson → Session | — | Provenance |
| `DECIDED_IN` | Decision → Session | — | Provenance |
| `EVOLVED_FROM` | Skill → Skill | `evolution_type (fix/derived/captured), diff` | Skill lineage DAG |
| `APPLIED_IN` | Skill → Session | `outcome, tokens_input, tokens_output, llm_calls` | Skill performance tracking |
| `LOCATED_IN` | Secret → File | — | Links secret to code file (Axon's File node) |

### 3.5 Temporal Model (inspired by Graphiti)

- `valid_at` / `invalid_at` on ALL memory nodes AND on SUPERSEDES, CONTRADICTS edges
- `created_at` / `updated_at` on all memory nodes
- Soft-delete via `invalid_at` timestamp — never hard-delete, preserves lineage
- **Query semantics (P2 fix):** All forge_* tools default to "current" view (`WHERE n.invalid_at IS NULL`). Pass `include_historical=true` to see invalidated nodes. `forge_timeline` always includes historical nodes (that's its purpose). Dangling edges to invalidated nodes are excluded from current-view traversals.

### 3.6 Cross-Graph Query Example

```cypher
// "Show me all decisions that affect the auth module"
MATCH (d:Decision)-[:AFFECTS]->(f:Function)
WHERE f.file_path CONTAINS 'auth' AND d.status = 'active'
RETURN d.title, d.rationale, f.name, f.file_path
```

## 4. MCP Tools

### 4.1 Code Intelligence (proxied from Axon — 15 tools, sandboxed)

```
axon_query, axon_context, axon_impact, axon_dead_code, axon_detect_changes,
axon_cypher, axon_communities, axon_coupling, axon_call_path, axon_cycles,
axon_file_context, axon_test_impact, axon_review_risk, axon_explain,
axon_list_repos
```

**SECURITY (P0 fix):** `axon_cypher` is sandboxed to READ-ONLY queries against CODE node types only. A query allowlist rejects any Cypher that references memory node labels (Decision, Pattern, Lesson, Preference, Session, Skill) or contains write keywords (CREATE, SET, DELETE, MERGE, REMOVE). This prevents bypass of forge_* tool guards via raw Cypher. Implementation: query parser validates AST before execution.

### 4.2 Memory Tools (new — 8 tools)

| Tool | Args | Returns | Path |
|------|------|---------|------|
| `forge_remember` | `type (decision/pattern/lesson/preference), content_or_structured, links_to[]` | `node_id` | Structured input = deterministic. Natural language = 1 LLM call. |
| `forge_recall` | `query, type?, timeframe?, limit?` | `node[]` | FTS first → vector second → LLM rerank only if >10 results |
| `forge_link` | `from_id, to_id, edge_type, properties?` | `edge_id` | Deterministic |
| `forge_decisions` | `code_path_or_symbol` | `decision[]` | Deterministic (Cypher traversal) |
| `forge_patterns` | `domain?, min_confidence?` | `pattern[]` | Deterministic (Cypher filter) |
| `forge_timeline` | `node_id_or_query, depth?` | `chain[]` | Deterministic (Cypher chain) |
| `forge_forget` | `node_id, reason` | `ok` | Deterministic (set invalid_at) |
| `forge_usage` | `session_id?, last_n_sessions?, group_by?` | `breakdown` | Deterministic (Cypher aggregation) |
| `forge_scan` | `path?, depth (shallow/full)` | `secret[]` | Deterministic (regex + entropy, zero LLM) |

**SECURITY (P1 fix):** ALL Cypher queries use parameterized queries (`$param` syntax) — never string interpolation. Tool args are typed (UUID for IDs, enum for edge_type, validated strings for paths). `forge_link` restricts `edge_type` to the allowlisted set from §3.4. `forge_recall` reranking redacts `Preference.value` and `Session.summary` fields before sending to LLM.

### 4.3 Deterministic vs Agent Path Map

**Deterministic (zero LLM):** Code indexing, forge_decisions, forge_patterns, forge_timeline, forge_link, forge_forget, forge_usage, PostToolUse decision flagging, session create/close, metric computation, evolution threshold checks, skill registration, file pattern matching, schema migration.

**Agent (LLM justified):**

| Operation | Why LLM needed | Guard |
|-----------|---------------|-------|
| `forge_remember` (natural language) | Extract structure from prose | Structured input path available as zero-token alternative |
| `forge_recall` (semantic) | Query understanding for reranking | FTS first, vector second, LLM ONLY if >10 results |
| AUTO-FIX | Diagnose failure + generate fix | User approval required |
| AUTO-IMPROVE | Generate improved skill | User approval required |
| AUTO-LEARN | Extract pattern from session | User confirms each candidate |
| SessionEnd decision extraction | Identify unrecorded decisions | Bounded to current session |

**Target:** >80% deterministic ratio, visible via `forge_usage`.

## 5. Self-Evolution Engine

### 5.1 Three Modes

**AUTO-FIX:** Triggered when `fallback_rate > 0.40` or skill fails in session. Detected at SessionEnd (deterministic metric check). Surfaced at next SessionStart. User picks from options → LLM generates fix in worktree (1 call) → user reviews diff → validated apply.

**AUTO-IMPROVE (DERIVED):** Triggered when `effective_rate < 0.55` with `applied_rate > 0.25`. Detected every 5th session. Surfaced at SessionStart. Same approval flow.

**AUTO-LEARN (CAPTURED):** Triggered when novel workflow detected at SessionEnd (no existing Pattern node matches with cosine similarity > 0.85). User confirms capture → LLM extracts pattern (1 call) → stored as Pattern node first → promoted to Skill only after user approval + one successful test run.

**Metric definitions (P2 fix):**
- `applied_rate` = `total_applied / total_selections` (how often a selected skill is actually used)
- `completion_rate` = `total_completions / total_applied` (how often an applied skill succeeds)
- `effective_rate` = `total_completions / total_selections` (end-to-end success)
- `fallback_rate` = `total_fallbacks / total_applied` (how often the skill falls back to manual)
- **Minimum support:** Metrics only trigger evolution when `total_selections >= 5` (cold-start guard)
- **Sample window:** Last 10 sessions or 30 days, whichever is smaller

### 5.2 Evolution Containment (P0 fix)

The evolution engine generates code and writes to disk. This is the highest-risk surface.

**Containment model:**
1. **Isolated worktree:** All evolution runs in a throwaway git worktree (`git worktree add /tmp/forge-evolve-$RANDOM`). Never modifies the working tree directly.
2. **Allowlisted paths:** Evolution can ONLY modify files matching `skills/**/*.md` and `skills/**/templates/*`. Hooks, plugin.json, scripts, agents, and any non-skill files are BLOCKED.
3. **Output sanitization:** Before writing ANY LLM output:
   - Strip `<think>...</think>` blocks (structured parser, not regex — handles nested tags)
   - Validate SKILL.md frontmatter against JSON schema (required fields: name, description)
   - Reject if output contains tool invocation syntax, shell commands, or URL patterns
   - Max file size: 50KB per skill file
4. **Validation gate:** After writing to worktree:
   - Frontmatter schema validation
   - ShellCheck on any referenced scripts (if applicable)
   - Diff size limit: max 500 lines changed per evolution
5. **Apply via merge:** Validated worktree changes are applied as a git patch (`git diff | git apply`) to the real tree. The worktree is deleted after.
6. **Rollback:** If apply fails or user rejects post-apply, `git checkout -- skills/` reverts. No git stash.
7. **Audit trail:** Every evolution attempt (success or failure) logged as a `Lesson` node with the diff, reason, and outcome.

### 5.3 OpenSpace Bugs Fixed

| OpenSpace Bug | Root Cause | Our Fix |
|--------------|-----------|---------|
| #41 `<think>` block corruption | No output sanitization | Regex strip + frontmatter validation before write |
| #36 Init hangs | Blocking LLM init on asyncio | Fully async init with 10s timeout |
| #11 ErrorCode enum as function | Bad pattern | Proper exception classes |
| #13 CLI flag crashes | Untested code path | CI tests all CLI paths |
| #15 Tool fallback never executes | Branch on wrong key | Branch on resolved object |
| #9 `"no" in "know"` | Substring matching | Word-boundary regex |
| #29 Non-idempotent registration | Returns None on existing | Cypher MERGE — always returns record |
| #40 59s skill search | No pagination | Graph-local only, no cloud fetch |
| #38 Routing confusion at scale | Flat catalog | Hierarchical: domain → type → skill |
| #19 Traceback leaks to MCP | format_exc in responses | Log server-side, generic errors to client |
| #31 Supply-chain deps | No upper bounds | Pin all dependency upper bounds |
| Race condition | No file locking | fcntl.flock + atomic temp-dir → rename |
| No rollback on crash | Orphaned state | git stash before, git stash pop on failure |

### 5.3 Evolution UX Flow

- **Detection:** Deterministic checkpoints (SessionStart, SessionEnd, every 5th session). Pure Cypher metric aggregation — zero tokens.
- **Surfacing:** Natural boundaries only (SessionStart/SessionEnd). Never mid-task.
- **User interaction:** Always explicit options (a/b/c/d). Nothing happens without a letter.
- **Execution:** Evidence gathering (deterministic) → LLM fix/improvement (1 call, tracked) → diff presented → user approves → atomic apply with git rollback safety.
- **Visibility:** Token cost shown after every evolution. Skill generation tracked in graph.

## 6. Token Tracking

**Per tool call** — every MCP response includes `_meta`:

```json
{
  "_meta": {
    "tokens_input": 0,
    "tokens_output": 0,
    "llm_calls": 0,
    "duration_ms": 42,

  }

```

**Per session** — aggregated on Session node: `total_tokens_input`, `total_tokens_output`, `total_llm_calls`, `total_tool_calls`, `deterministic_ratio`.

**Per skill** — on APPLIED_IN edges: `tokens_input`, `tokens_output`, `llm_calls`.

### 6.2 Usage Tool

`forge_usage` returns token breakdown grouped by skill, tool, or session. Shows deterministic ratio trend over time.

## 7. Forge HUD (StatusLine Dashboard)

### 7.1 Mechanism

Uses Claude Code's native `statusLine` API. Configured in `~/.claude/settings.json`:
```json
{
  "statusLine": {
    "type": "command",
    "command": "node ${CLAUDE_PLUGIN_ROOT}/forge-hud/dist/index.js"
  }
}
```

Rust static binary (~2MB). Short-lived process: reads stdin JSON + hud-state.json → renders ANSI text to stdout → exits. <2ms per cycle. Polled by Claude Code ~300ms.

### 7.2 Display

```
Line 1: 🔨 forge v0.2.0 │ 📊 1,247 nodes · 3,891 edges │ 🧠 23 decisions · 8 patterns │ ⚡ feature · build
Line 2: 💰 12.4K/8.2K tokens (87% det) │ 🔄 6 skills · 1 fix │ 👥 planner ✓ · generator ⚙ 2/4 · evaluator ⏳
```

Adapts to terminal width. Collapses progressively at 120 and 80 columns.

### 7.3 Data Flow

forge-graph MCP server writes `hud-state.json` on every tool call. HUD reads it. One-way. <1ms read.

```json

  "graph": { "nodes": 1247, "edges": 3891 },
```

`--include-hud-features` flag replicates claude-hud essentials (model, context%, git) so users don't need both plugins.

## 8. Agent Observability

### 8.1 MCP-Level Tracking

All Forge agents call forge-graph tools. Every MCP call updates hud-state.json with: agent identity, last tool called, current file, token count, duration.

Agent identification via `agent_id` parameter on every tool call.

### 8.2 Task-Level Tracking

HUD polls `~/.claude/tasks/` for shared task list. Shows per-agent task ownership and status.

### 8.3 Transcript-Level Tracking

For non-MCP tool calls (Read, Edit, Bash), discover subagent transcript files by scanning `~/.claude/projects/` for recent JSONL. Parse for tool_use events. ~1s delay.

### 8.4 Drill-Down Skill

`/forge:agents` skill queries hud-state.json and formats detailed per-agent view as markdown in conversation.

## 9. Hook Integration

| Hook | Trigger | What It Does |
|------|---------|-------------|
| **SessionStart** | Session begins | Create Session node. Load relevant decisions/patterns/lessons. Check skill health. Inject context via hookSpecificOutput. |
| **PostToolUse** (Edit/Write) | File edited | Check if file has AFFECTS edges to active Decisions. Flag if so. Async. |
| **TaskCompleted** | Task finished | Record outcome on Session node. Update APPLIED_IN edges. |
| **SessionEnd** | Session closes | **Drain barrier:** wait for pending async PostToolUse writes (max 5s timeout). Then: close Session node, scan for unrecorded decisions (user-gated), compute skill metrics, queue evolution suggestions. |

## 10. Security & Trust Model

### 10.1 Trust Levels

| Data Source | Trust Level | Treatment |
|------------|-------------|-----------|
| User-confirmed decisions | HIGH | Inject into context as-is |
| Agent-detected decisions | LOW | Sanitize: strip instructions, tool syntax, URLs. Inject as summaries only. Require user confirmation to promote to HIGH. |
| Preferences | SENSITIVE | Never send to LLM reranking. Redact `value` field in forge_recall results sent to LLM. |
| Session summaries | SENSITIVE | Never send to external LLM. Redact in forge_recall LLM reranking. |
| Axon code nodes | HIGH | Deterministic (tree-sitter parsed). Trusted. |
| Transcript content | UNTRUSTED | Never promote into model context without sanitization. Parse with strict JSON schema. Reject malformed entries. |
| HUD state file | INTERNAL | Not user-facing data. Operational only. |

### 10.2 Authorization Model (P1 fix)

| Agent | Allowed Tools | Scope |
|-------|--------------|-------|
| **Planner** | axon_query, axon_context, axon_impact, forge_recall, forge_decisions, forge_patterns | READ-ONLY on all node types |
| **Generator** | axon_query, axon_context, axon_file_context, forge_recall, forge_decisions | READ-ONLY. No memory writes. |
| **Evaluator** | axon_query, axon_context, axon_impact, axon_review_risk, forge_recall | READ-ONLY. No memory writes. |
| **Lead (user session)** | ALL tools | Full access |
| **Evolution engine** | forge_remember, forge_link (internal only) | Write ONLY to Skill, Pattern nodes. Restricted to worktree for file writes. |

Enforced at the MCP server level via `agent_id` on every request. Unknown agent IDs default to READ-ONLY.

### 10.3 Graph Isolation

Axon and Forge share one LadybugDB file but operate on disjoint namespaces:
- **Axon owns:** File, Folder, Function, Class, Method, Interface, TypeAlias, Enum, Community, Process + all code edges
- **Forge owns:** Decision, Pattern, Lesson, Preference, Session, Skill + all memory edges
- **Cross-namespace edges** (AFFECTS, DECIDED_IN connecting memory → code): owned by Forge, READ by Axon
- **Transaction model:** Axon indexes are bulk writes (periodic). Forge memory writes are individual transactions. LadybugDB supports concurrent readers with single writer. Indexing and memory writes are serialized via a write lock with 5s timeout.
- **Schema versioning:** `_forge_meta` table stores schema version. On startup, forge-graph checks version, runs migrations if needed. Migrations are sequential, reversible (down migrations defined for each up), and preflight-checked against current schema before execution. Backup snapshot (`.lbdb.bak`) created before any migration.

### 10.4 HUD File Security (P1 fix)

- `hud-state.json` stored in `${CLAUDE_PLUGIN_DATA}/hud/` (fixed directory, not configurable)
- Written with `O_NOFOLLOW` flag — rejects symlinks
- Permissions set to `0600` (owner-only read/write)
- Parent directory canonicalized via `os.path.realpath()` and verified as prefix of `CLAUDE_PLUGIN_DATA`
- File paths in agent state are relative (not absolute) — no information leak about directory structure
- **Write debounce (P2 fix):** Max 1 write per 100ms. Coalesce rapid MCP calls. Reduces SSD churn.

### 10.5 Transcript & Task Scanning Security (P1 fix)

- Transcript scanning is **opt-in** (disabled by default, enabled via `forge_setup` config)
- When enabled: only scans `~/.claude/projects/${current_project}/` — not all projects
- Rejects symlinks (`os.path.islink()` check)
- Max file age: 24 hours. Max file size: 10MB. Older/larger files ignored.
- Strict JSON schema validation: only extracts `tool_use` events with known tool names
- **Never promotes transcript text into model context** — only extracts structured metadata (tool name, file path, duration)

### 10.6 Memory Injection Safety (P1 fix)

When SessionStart loads context from the graph:
1. Filter to `trust_level = 'user'` by default. Agent-trust nodes only included if user has opted in.
2. Strip any content matching: tool invocation patterns (`<tool_use>`), shell command syntax (`$ ...`, `` ` ``), URL patterns (`http[s]://`)
3. Inject as structured summaries (title + rationale), not raw text blocks
4. Cap: max 10 decisions, 5 patterns, 5 lessons per session injection — prevents context flooding

### 10.7 Adversarial Test Plan (P2 fix)

Release-blocking adversarial tests:
- **Cypher injection:** Attempt `'; DROP TABLE Decision; --` via forge_recall query param
- **axon_cypher memory bypass:** Attempt `MATCH (d:Decision) RETURN d` via axon_cypher
- **HUD symlink:** Create symlink at hud-state.json path → verify rejection
- **Transcript poisoning:** Craft malicious JSONL with prompt injection text → verify sanitization
- **Evolution path traversal:** Attempt evolution that writes outside skills/ → verify block
- **Concurrent hook race:** Trigger PostToolUse and SessionEnd simultaneously → verify no data loss
- **Oversized evolution:** Generate 1000-line skill diff → verify rejection
- **Memory exfil via rerank:** Store sensitive preference → verify redaction in LLM rerank call

### 11.5 Migration from v0.1.5 (P2 fix)

**Migration is a first-class operation, not a side effect.**

1. `forge-migrate` command (or auto-detected on first run):
   a. **Inventory:** Report what exists: SQLite graph files, MEMORY.md content, STATE.md sessions
   b. **Backup:** Copy all existing data to `${CLAUDE_PLUGIN_DATA}/v0.1.5-backup/`
   c. **Dry run:** Show what will be imported and what will be skipped
   d. **Import:** Convert MEMORY.md decisions to Decision nodes via structured `forge_remember` (deterministic path, zero LLM)
   e. **Verify:** Count imported nodes, report success/failure
   f. **Rollback:** If migration fails, restore from backup automatically
2. Old SQLite files preserved in backup (not deleted)
3. First Axon indexing runs after migration completes

### 11.6 Capability Negotiation (P2 fix)

Binary and pip installs have different capabilities:

| Capability | pip | binary |
|-----------|-----|--------|
| FTS search | Yes | Yes |
| Vector search (HNSW) | Yes | No (fastembed not bundled) |
| LLM reranking | Yes | Yes |
| Embeddings generation | Yes | No |

forge-graph exposes `forge_capabilities` resource listing what's available. Tools gracefully degrade: `forge_recall` skips vector step if embeddings unavailable, uses FTS + LLM rerank only. HUD shows capability mode.

## 12. Performance Requirements

| Operation | Target (p95) | Benchmark Source | Notes |
|-----------|-------------|-----------------|-------|
| Graph query (2-hop) | <10ms | LadybugDB: 6.6ms (100K nodes) | Warm cache. Cold cache: <50ms. |
| Graph ingestion (DB insert only) | <1s | LadybugDB: 0.58s (100K nodes + 2.4M edges) | DB write only. Full Axon pipeline (parse+resolve+insert): TBD, benchmark on representative repos. |
| FTS query | <5ms | LadybugDB built-in FTS | |
| Vector search (100K) | <20ms | LadybugDB HNSW | pip mode only |
| HUD state read + render | <5ms end-to-end | Rust binary: ~1ms startup + ~1ms JSON parse + ~1ms render | Benchmarked via `hyperfine`. |
| MCP tool response (deterministic) | <50ms | Cypher + serialize | |
| hud-state.json write | <2ms | Debounced: max 1 write per 100ms | |

**Server health (P2 fix):**
- **Memory:** Max heap tracked via periodic `resource.getrusage()`. Alert in HUD if >500MB.
- **Compaction:** LadybugDB auto-compacts. We trigger manual compaction every 100 sessions.
- **Restart policy:** MCP server is a child process of Claude Code — restarts when Claude Code restarts. On unhandled exception: log, flush HUD state, exit cleanly (Claude Code will relaunch).
- **Soak test in CI:** 1000 sequential tool calls, verify no memory growth >10%.

**Degraded mode (P1 fix):**
If LadybugDB fails to open (corruption, lock timeout):
- Code intelligence: falls back to Serena if available, else tools return graceful errors
- Memory tools: return empty results with `"degraded": true` in `_meta`
- HUD: shows "GRAPH OFFLINE" indicator
- Evolution: disabled entirely
- Session tracking: falls back to local JSON file (not graph)

This ensures a DB failure doesn't take out the entire plugin.
