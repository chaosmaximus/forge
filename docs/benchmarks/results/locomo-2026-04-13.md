## Wave 1 hybrid update (2026-04-14)

Wave 1 of the [improvement roadmap](../improvement-roadmap-2026-04-13.md) added a BM25 leg to the raw layer and fuses it with the existing KNN leg via pure Reciprocal Rank Fusion (k=60). The new `--raw-mode hybrid` is the default; `--raw-mode knn` preserves the 2026-04-13 baseline byte-for-byte.

### Implication for Wave 2 (bge-large)

Wave 2 of the roadmap plans to swap MiniLM for `bge-large-en-v1.5` (1024-dim), projected to add +2.8 pp to LoCoMo R@10 toward MemPalace's 0.924 bge-large hybrid baseline. **Wave 1 hybrid already scored 0.9456, overshooting MemPalace's bge-large number by +2.2 pp.** Wave 2 is not obsolete — bge-large may still improve LongMemEval, and semantic retrieval generally benefits from larger embedders — but the LoCoMo-specific headline lands before Wave 2 even runs. The wave ordering may be rethought after the full Wave 1 numbers are reviewed.

## Setup

| | |
|---|---|
| Samples | 10 conversations (`conv-26` … `conv-50`) |
| QA pairs | 1,986 |
| Sessions per conversation | 19–32 |
| Granularity | Session — chunks deduped to `session_<N>` at scoring time |
| Top-K retrieved | 50 chunks |
| Cosine cutoff | None (set to 2.0 to disable filtering) |
| Chunker | 800-char window, 100 overlap, 50-char minimum |

The bench builds **one** in-memory SQLite per sample, ingests every session as a single document containing all turns from both speakers (joined with newlines, prefixed with the speaker name), then runs every QA against the same corpus. Per-QA elapsed times are dominated by the embed of the question text — ~10 ms per question on M1 Pro single-core.

**Two metric families are mixed in this table.** Treat them carefully:
- **R@K** (this run, MemPalace) = retrieval recall at session granularity. Pure retrieval quality.
- **LLM-judge accuracy** (LoCoMo paper, Memori, Mem0, LangMem) = an LLM grades the system's full answer text against ground truth.

The two are NOT directly comparable. A system can have perfect retrieval (R@K=1.0) and fail at LLM-judge if its reader generates wrong answers from correct evidence; conversely, a system can succeed at LLM-judge with weak retrieval if the reader is good at filling in gaps. We report retrieval-only because Forge raw doesn't have a reader stage yet — adding LLM-judge accuracy is on the roadmap (see [plan.md](../plan.md) §3).

## Honest limitations

1. **Methodological difference vs MemPalace's baseline.** MemPalace's "session, no rerank, top-10" baseline (60.3%) treats each session as a single document. Forge raw chunks each session into 800-char windows, retrieves top-50 chunks, then dedupes by session_id. With ~25 sessions × ~5 chunks per session = ~125 chunks per corpus, the top-50 search depth covers a substantial fraction of the corpus before the dedupe collapses it back to session granularity. This is **part of why we sit so much higher than MemPalace's baseline**, and not a moat — anyone could change MemPalace's bench script to chunk + dedupe and recover most of those 27 points. We documented the difference rather than claim it as an architectural win.
2. **Top-50 search depth is wide.** A LoCoMo conversation has at most ~32 sessions and ~160 chunks. With top-K=50, our search effectively covers >30% of the corpus per query. A future run with `--top-k 10` (closer to a "tight" recall test) would produce a meaningfully different number — we'll publish that variant separately.
3. **Speaker prefixes inside session bodies.** We prefix each turn with its speaker name (`"Alice: hello"`). This is the same as MemPalace's LoCoMo bench script but differs from our LongMemEval bench (which uses user turns only with no prefix). Speaker prefixes may help category 5 (adversarial speaker confusion).
4. **No reader stage.** This is a retrieval-only number. We don't run an LLM reader to generate answers from the retrieved chunks, so we can't report end-to-end answer accuracy. Plan §3 calls this out as a planned addition.
5. **Single-machine, single-run.** Numbers from one execution on one machine — not averaged over multiple runs. The variance is small (the dedupe at session granularity is deterministic given a fixed model and chunker), but reproducibility on different hardware will need its own validation.
6. **Embedder and chunker are the MemPalace defaults.** We did not tune. Changing to bge-large (~+3pts on this task) or running a hybrid keyword + vector search (~+2-5 pts based on MemPalace's progression) are explicit follow-ups.

# Extract mode comparison (LoCoMo, full 1,986 QAs)

**Update:** after the raw mode publication above, we ran LoCoMo through Forge's extract mode — the 8-layer extraction pipeline via Gemini 2.5 Flash, storing memories in the Forge `memory` table with session_id stamped, retrieved via BM25. Same 10 conversations, same 1,986 QAs, same granularity.

## Results

| Metric | Raw | Extract (Gemini 2.5 Flash) | Delta |
|---|---:|---:|---:|
| Mean Recall@5 | **0.7638** | 0.5937 | **−17.0 pp** |
| Mean Recall@10 | **0.8746** | 0.7090 | **−16.6 pp** |
| Mean NDCG@10 | **0.6437** | 0.5273 | −11.6 pp |
| Wall time | 33.79 s | 505.66 s | 15× slower |

## Per-category Recall@10

| Category | Raw | Extract | Delta |
|---|---:|---:|---:|
| 1 — Single-hop | 0.9291 | 0.8262 | −10.3 pp |
| 2 — Temporal | 0.8474 | 0.6760 | **−17.1 pp** |
| 3 — Temporal-inference | 0.7604 | 0.6354 | −12.5 pp |
| 4 — Open-domain | 0.8775 | 0.7099 | **−16.8 pp** |
| 5 — Adversarial (speaker confusion) | 0.8789 | 0.6726 | **−20.6 pp** |

Extraction loses across **every** category. Biggest losses are in adversarial (−20.6 pp) and temporal/open-domain (both ~−17 pp). Smallest loss is on single-hop (−10.3 pp) — simple fact recall where Gemini's extracted memory content is closest to the query vocabulary.

## Consistency with LongMemEval

The LongMemEval 50-question comparison (see `longmemeval-2026-04-13.md` §"Four-mode comparison") shows the same pattern: **extract loses ~18 pp R@5** to raw. LoCoMo's ~17 pp gap is statistically indistinguishable from LongMemEval's. The finding is **benchmark-invariant**: Forge's current extraction prompt consistently loses ~17–18 percentage points of retrieval recall compared to raw storage of the underlying text.

## Why the adversarial category shows the biggest loss

LoCoMo category 5 is designed to test **speaker confusion** — the question references one speaker's statement, but the corpus contains similar text from the other speaker. Raw mode's chunks preserve the literal speaker prefix (`Alice: ...` / `Bob: ...`) so BM25 / vector similarity can disambiguate. Forge's extraction prompt summarizes turns into structured memories and drops the speaker information in many cases (the prompt is not speaker-aware), so the extracted memories from both speakers become indistinguishable.

This is a fixable design issue — adding speaker attribution to the extraction prompt would likely close most of the adversarial gap. It is not, however, a free fix: a speaker-aware extraction prompt becomes conversation-specific and loses transferability.
